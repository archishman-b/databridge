Foundations

The access gap

Why strategy professionals have always needed an intermediary to answer their own questions

Every meaningful commercial dataset — customer transaction history, discount by rep, revenue by SKU, retention by cohort — exists as rows and columns in a table somewhere. The question is almost never whether the data exists. It is whether the person who knows what question to ask can also run the analysis that answers it.

In a pricing transformation engagement at Monitor Deloitte, the most consequential analyses were not technically complex. They were: who is giving away the most discount? Which customers are receiving the worst pocket price? Which reps are most likely to break from price policy under pressure? These are straightforward aggregations and segment comparisons. But they depended on a cleaned, joined, deduplicated dataset sitting in SQL Server — and extracting it required an analyst queue, a data engineering team, and a 48-hour turnaround on every iteration.

The strategic insight did not require machine learning. It required filter, group by, and sort. The bottleneck was never analytical sophistication. It was infrastructure setup time, analyst queue lag, and the structural separation between the person who understood the business question and the person who could run the query.

The analysis that changed the pricing strategy was a grouped aggregation on three columns. It took an analyst four hours to produce the first iteration. DataBridge runs it in four seconds.

— Pricing transformation, US auto parts retail, Monitor Deloitte

DataBridge is not a replacement for SQL expertise or data engineering. It is a direct path from raw data to strategic answer for the analyst-consultant hybrid who has the business question clearly in their head and needs to move quickly.

Foundations

What data actually is

Rows, columns, types, and the shape of a useful dataset

Every dataset DataBridge works with follows the same structure: a rectangular table where each row is one observation (a transaction, a customer, a product) and each column is one attribute of that observation. This is called tidy data, and almost every analytical operation in the world assumes it.

📋
Rows = observations
One row is one instance of the thing being measured. One transaction, one customer, one survey response, one SKU at one time point.
⬜
Columns = attributes
Each column describes one property of the observation. Revenue, region, date, customer tier, discount percentage.
123
Types matter
Numbers, text, and dates behave differently. You cannot average a text column or filter a date with string logic. DataBridge infers and displays types automatically.
∅
Nulls are real
A missing value (null) is not zero. Averaging over nulls gives a different result than averaging over zeros. Understanding null distribution is part of understanding the data.

Most raw datasets arriving from ERP systems, CRMs, or data warehouse exports are not tidy. They have mixed types (a revenue column with some text entries), structural nulls (deals without a close date because they never closed), and implicit joins (a customer ID that needs to be matched to a customer master table before you can do any segmentation). The first step in any analysis is getting the data into a state where your operations will produce reliable results.

The pipeline model

Why operations are chained, not run in isolation

DataBridge uses a pipeline architecture: every operation you apply transforms the current state of the data and passes it to the next step. The result of step 3 feeds step 4. This is how professional data tools like dbt, Pandas, and SQL CTEs work — and for good reason.

Raw transactions
›
Join customer master
›
Filter: Won deals only
›
Group by rep + region
›
Pricing diagnostic

Each step in the pipeline is recorded in the audit log and can be undone independently. This lets you explore multiple analytical paths from the same source data without losing your work. You can also save intermediate results as named tables in the sidebar, which allows branching — taking the same cleaned dataset in two different analytical directions simultaneously.

The pipeline model also makes the analysis auditable. A senior partner or client can follow the exact sequence of transformations that produced the output. There is no hidden formula cell, no macro, no "I ran it through the model." Every step is explicit and reversible.

In SQL terms: each pipeline step is equivalent to one CTE (Common Table Expression) in a WITH clause. The final result is what a SELECT * FROM the last CTE would return. DataBridge's pipeline is visual SQL.
Operations

Joins — combining tables

The most consequential operation in commercial analytics

Almost no real analysis runs on a single flat table. A transaction table has customer IDs but not customer names or tiers. A CRM export has deal values but not product categories. A rep performance table has revenue but not territory. Joins are how you bring the right context into the right row.

The core idea is simple: find rows in two tables that share a common key value, and combine their columns into one row. But the choice of join type determines what you keep when the keys don't perfectly match — which is most of the time in real commercial data.

Join typeWhat it keepsWhen to use it
Inner joinOnly rows where the key exists in both tablesWhen you only care about matched records — e.g. transactions with a known customer
Left joinAll rows from the left table; null for unmatched right columnsMost common in strategy work. Keep all transactions, add customer attributes where available
Right joinAll rows from the right table; null for unmatched left columnsFind which customers have no transactions — useful for lapsed account analysis
Full outerAll rows from both tables, nulls where unmatchedReconciliation — find gaps in both directions simultaneously
Anti-joinRows in the left table with no match in the rightFind orphan records — transactions with no customer master, leads with no account
LookupAll left rows, one additional column from the rightEquivalent to VLOOKUP — bring in customer tier, territory name, manager, etc.

The match rate of a join is diagnostic in itself. If you left join 10,000 transactions to a customer master and 2,000 rows have null customer names, that tells you 20% of your transactions have no master record — a data quality issue that will distort every subsequent segmentation and aggregation.

-- Left join: all transactions, customer attributes where available SELECT t.*, c.customer_name, c.tier, c.account_manager FROM transactions t LEFT JOIN customer_master c ON t.customer_id = c.customer_id;
Operations

Filter & select

Scoping the dataset to the population that answers your question

Every analytical question has a scope. "Which enterprise accounts are at risk of churn?" requires filtering to enterprise tier before any further analysis makes sense. "What is the average discount on electronics deals?" requires filtering to category = Electronics first. Filtering is not data reduction — it is question definition.

DataBridge's filter builder supports compound conditions (AND / OR logic across multiple columns), text matching (contains, starts with), numeric comparisons (greater than, less than, between), and null detection (is empty / is not empty). These cover the overwhelming majority of real scoping needs in commercial analytics.

-- Compound filter: won deals in the North region above $5,000 SELECT * FROM transactions WHERE status = 'Won' AND region = 'North' AND revenue > 5000;

Column selection (keeping or dropping columns) is not just cosmetic. Wide datasets slow down every subsequent operation and make output tables hard to read. A disciplined approach to column selection — keeping only what the analysis needs — is a hallmark of clear analytical thinking.

Deduplication is one of the most underused operations in commercial analytics. CRM exports frequently contain duplicate deal records from multiple pipeline stages. Customer master tables may have duplicate IDs from system migrations. Running a deduplication check before any aggregation prevents inflated counts and incorrect revenue figures.

Operations

Aggregate & group

Collapsing rows into summaries — the heart of most strategic diagnostics

Aggregation is the operation that turns a list of 10,000 transactions into a 5-row summary by region. It is the most frequently used operation in commercial analytics, and it is the one most commonly done incorrectly in spreadsheets — because spreadsheets encourage manual construction of aggregations rather than declarative grouping.

The GROUP BY pattern is: choose the dimensions you want to group by (region, rep, product category, customer tier), then choose what to measure within each group (total revenue, average discount, count of deals, number of unique customers). Every strategic dashboard is some variant of this pattern.

FunctionWhat it computesStrategy use case
SUMTotal of all values in the groupTotal revenue by region, total discount by rep
COUNTNumber of rows in the groupNumber of deals by stage, number of transactions by customer
COUNT DISTINCTNumber of unique valuesNumber of unique customers per rep, unique SKUs per account
AVGMean value within the groupAverage deal size by tier, average discount % by product
MIN / MAXSmallest or largest valueFloor and ceiling price by customer, date of first / last transaction
MEDIANMiddle value (not affected by outliers)Median discount — more robust than mean when there are outlier deals

The choice between AVG and MEDIAN matters more than most strategy practitioners realise. In a deal dataset where one account received a 60% discount and the rest received 10–15%, the mean discount for that group will be significantly inflated. The median is immune to this distortion. For pricing diagnostics, median pocket price and median discount are almost always more meaningful than averages.

Pivot tables are a special form of aggregation where the values of one column become the column headers of the result — for example, turning a transaction table into a matrix of revenue by region (rows) × product category (columns). Pivots are powerful for portfolio analysis and cross-dimensional diagnostic views.

-- Classic pricing diagnostic: avg and median discount by sales rep SELECT sales_rep, COUNT(*) AS deal_count, AVG(discount_pct) AS avg_discount, PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY discount_pct) AS median_discount, SUM(revenue) AS total_revenue FROM transactions WHERE status = 'Won' GROUP BY sales_rep ORDER BY avg_discount DESC;
Operations

Transform columns

Creating derived metrics and reshaping data for analysis

Raw datasets rarely contain the exact columns your analysis needs. Pocket price must be derived from list price and discount percentage. Margin must be derived from revenue and cost. Customer tenure must be derived from onboarding date and today's date. Calculated columns are where analytical insight lives — they encode business logic into the data itself.

ƒ
Arithmetic
Combine two columns with +, −, ×, ÷. Derive margin %, pocket price, revenue per unit, discount amount from existing columns.
IF
Conditional logic
IF column meets condition THEN value A ELSE value B. Flag deals above policy, tag reps as compliant or non-compliant, mark accounts as at-risk.
Aa
Text operations
Trim, standardise case, concatenate, split, extract characters. Essential for cleaning CRM data where rep names or region codes are inconsistently formatted.
📅
Date extraction
Extract year, quarter, month, week, weekday from any date column. Enables period-based aggregations without manual calendar manipulation.

Binning turns a continuous numeric column into discrete buckets — converting deal sizes into Small / Mid / Large / Enterprise buckets, for example, or discount percentages into 0–5% / 5–10% / 10–20% / 20%+ bands. Binned data is far more readable in pivot tables and segment analyses than raw numeric distributions.

Ranking is essential for rep performance analysis, customer value ranking, and product contribution sorting. Dense rank (no gaps in rank numbers when there are ties) is usually preferable to standard rank for league table presentations.

Inspect & profile

Understanding the data before you trust the analysis

A data profiling step before any substantive analysis is not optional — it is the difference between a diagnostic that is reliable and one that presents confidently incorrect conclusions. Every experienced data analyst profiles the data before they analyse it. Every experienced strategy consultant who has been burned by data quality problems does the same.

OperationWhat it reveals
Column summaryMin, max, mean, median, distinct count, null % for every column. The fastest way to understand a new dataset.
Null auditCount and % of missing values per column. Informs which columns can be reliably used and which carry data quality risk.
Duplicate finderRows that appear more than once, optionally by key columns. Prevents double-counting in aggregations.
Value frequenciesTop N most common values in a column. Reveals category distributions, flags unexpected values, identifies data entry inconsistencies.
CrosstabTwo-way frequency table. Classic diagnostic for understanding the relationship between two categorical variables.
Correlation matrixPairwise correlation between numeric columns. Reveals which metrics move together — discount and deal size, tenure and churn, price and volume.
Pareto / 80-20Cumulative contribution sorted descending. Instantly identifies which customers, products, or reps drive the majority of revenue or discount leakage.
The Pareto operation is particularly powerful in pricing diagnostics. Sorting customers by total discount received descending and showing cumulative percentage almost always reveals that 10–15% of accounts receive 60–70% of all discount value. This is the starting point for a discount governance intervention.
Operations

Segment & classify

Assigning rows to categories based on business logic

Segmentation is the process of assigning each row to a category based on one or more of its attributes. Almost all strategic frameworks reduce to a segmentation: accounts are High / Mid / Low value, deals are Above / Below policy, customers are New / Growth / At-risk / Lapsed. The question is always which attributes drive the assignment and where the boundaries are.

Conditional tagging implements business-logic segmentation directly: define a sequence of rules (IF discount_pct > 20 THEN "High Risk", IF discount_pct > 10 THEN "Watch", ELSE "Compliant"), apply them top-to-bottom, and assign the first matching label. This produces a new column that can drive all subsequent aggregations and visualisations.

Percentile bucketing produces data-driven segmentation. Quartile (Q1–Q4) and decile (D1–D10) assignments divide the dataset into equal-count groups based on the distribution of one metric. Decile analysis on customer revenue is a standard starting point for enterprise account prioritisation — the top decile typically accounts for 50–60% of revenue in most B2B portfolios.

Time intelligence

Turning timestamped data into trend and period analysis

Most commercial datasets are timestamped, but timestamps alone do not produce trend analysis. You need to extract period dimensions from dates (year, quarter, month), compare the same period across different time horizons (year-over-year, month-over-month), and calculate trailing averages to smooth out noise. This is what time intelligence means.

OperationWhat it enables
Date partsExtract Year, Quarter, Month, Week, Weekday from a date column. Enables GROUP BY period without manual calendar lookup.
Fiscal calendarMap calendar dates to a fiscal year starting any month. Essential when your organisation reports on a non-January fiscal year.
Period comparisonAdd columns for prior period value, delta, and % change. YoY, MoM, QoQ, WoW — the standard period-over-period view in any executive dashboard.
Rolling averageTrailing N-period average. Smooths weekly noise in revenue data to reveal the underlying trend. Standard in sales operations reporting.
Running totalCumulative sum within a group, sorted by date. Tracks year-to-date performance against target, or cumulative discount spend against budget.
Lag / LeadPrevious or next row value within a group. Compares each transaction against the same customer's prior transaction — useful for detecting price erosion patterns.
-- Year-over-year revenue comparison using window functions SELECT *, LAG(monthly_revenue, 12) OVER (PARTITION BY region ORDER BY year_month) AS prior_year_revenue, monthly_revenue - LAG(monthly_revenue, 12) OVER (PARTITION BY region ORDER BY year_month) AS yoy_delta FROM monthly_revenue_by_region;

SQL primer

What every strategy professional should understand about the language that runs commercial analytics

SQL (Structured Query Language) is the language that almost all commercial data operations ultimately run on. It is not a programming language — it is a declarative specification: you describe what you want, and the database engine decides how to produce it. This is why SQL is learnable for non-engineers in a way that Python or R are not.

Every DataBridge operation has an exact SQL equivalent, shown via the "View equivalent SQL" toggle in each modal. This is intentional — DataBridge is partly a learning tool that makes SQL readable in context, without requiring you to write it.

SELECT
SELECT — what columns
Specify which columns to return, and any derived expressions. SELECT * returns everything. SELECT col1, col2 returns only those columns.
FROM
FROM — which table
The source table. The JOIN clause extends FROM to combine multiple tables on a key column.
WHERE
WHERE — which rows
Filter conditions. Only rows matching the WHERE clause are returned. Equivalent to DataBridge's Filter operation.
GROUP
GROUP BY — aggregation
Collapse rows into groups. Every column in SELECT that is not an aggregate function must appear in GROUP BY.
ORDER
ORDER BY — sort
Sort the result by one or more columns, ascending (ASC) or descending (DESC). LIMIT N returns the top N rows.
OVER
OVER — window functions
Compute a value across a set of rows related to the current row — without collapsing them. Running totals, ranks, lag/lead, rolling averages are all window functions.
-- The anatomy of a complete SQL query SELECT region, sales_rep, SUM(revenue) AS total_revenue, -- aggregate AVG(discount_pct) AS avg_discount, -- aggregate COUNT(*) AS deal_count -- aggregate FROM transactions -- source WHERE status = 'Won' -- filter rows first GROUP BY region, sales_rep -- then group HAVING SUM(revenue) > 10000 -- filter on aggregated result ORDER BY total_revenue DESC -- sort output LIMIT 20; -- top 20 only

Common patterns

The five SQL constructs that appear in 90% of commercial analytics work

1. The segmentation CTE — build a derived column in a WITH clause, then filter or aggregate on it downstream. This is the SQL equivalent of DataBridge's pipeline: transform first, analyse second.

WITH tagged_deals AS ( SELECT *, CASE WHEN discount_pct > 20 THEN 'High Risk' WHEN discount_pct > 10 THEN 'Watch' ELSE 'Compliant' END AS compliance_tag FROM transactions ) SELECT compliance_tag, COUNT(*), SUM(revenue) FROM tagged_deals GROUP BY compliance_tag;

2. The self-join for period comparison — joining a table to itself on a different period to compare current vs. prior performance.

SELECT curr.region, curr.revenue AS this_year, prev.revenue AS last_year, (curr.revenue - prev.revenue) / prev.revenue * 100 AS yoy_pct FROM monthly_revenue curr JOIN monthly_revenue prev ON curr.region = prev.region AND curr.year = prev.year + 1;
DataBridge's Period Comparison operation automates this pattern. Understanding it as a self-join makes the logic transparent when you need to adapt it.

Data quality

What to look for before trusting any analysis

DataBridge scores data quality automatically on load — but understanding what the score measures is more valuable than the score itself. The four dimensions that matter most in commercial analytics:

DimensionWhat to checkImpact if ignored
CompletenessNull % per column. Any analysis column with >10% nulls needs scrutiny.Aggregations silently exclude null rows, producing understated counts and means.
UniquenessDuplicate row check on key columns (transaction ID, customer ID).Aggregations double-count. Revenue totals are inflated. Trend analysis shows false spikes.
ConsistencyMixed types in numeric columns. "1,200" vs 1200. "N/A" in a revenue field.Averages and sums fail silently or return null. Filters on numeric ranges miss text-formatted values.
ValidityNegative values in price/revenue columns. Future dates in transaction history. Discount > 100%.Margin calculations produce impossible results. Period comparisons are distorted by erroneous anchors.

Schema design

Why the pricing schema mapper exists and what it enables

A schema is a named mapping between column names in your data and semantic roles in an analytical framework. The same column might be called "unit_price", "list_price", "sale_price", "price_usd", or "Price" across different exports from the same CRM. Without a schema, every tool that tries to understand what the column means has to guess.

DataBridge's pricing schema mapper defines 12 standard dimensions — list price, pocket price, unit cost, discount %, quantity, sales rep, customer, tier, region, product, and date — and lets you map your columns to them. This mapping is stored and passed to the Pricing Analyser when you hand off the dataset, so the Pricing Analyser already knows which column is the discount percentage and which is the unit cost.

This is the same concept as a semantic layer in enterprise data stacks (dbt metrics, Looker's LookML, Tableau's data model). The schema mapper brings a lightweight version of this concept to the browser, without any infrastructure.

Glossary

Key terms used throughout DataBridge and this companion

Tidy data
A dataset where each row is one observation and each column is one attribute. The standard input format for all DataBridge operations.
Pipeline
A sequence of operations applied in order, where each step's output becomes the next step's input. DataBridge records every step and allows undo at any point.
Join
An operation that combines rows from two tables based on matching values in a key column.
Aggregation
Collapsing multiple rows into summary statistics (SUM, COUNT, AVG, etc.) grouped by one or more dimensions.
Window function
A SQL computation that operates across a set of rows related to the current row without collapsing them. Running total, rank, lag, and rolling average are all window functions.
CTE
Common Table Expression. A named temporary result in a SQL query defined with WITH. Equivalent to one step in a DataBridge pipeline.
Null
A missing or unknown value. Not the same as zero. Most aggregation functions ignore nulls, which can silently distort results.
Schema
A named mapping between column names and semantic roles. DataBridge's pricing schema mapper defines which column is the discount %, which is the customer tier, etc.
Pocket price
The actual price received after all discounts, rebates, and allowances are applied. The most economically meaningful price in a waterfall analysis.
Cardinality
The number of distinct values in a column. High-cardinality columns (customer ID, transaction ID) are usually keys. Low-cardinality columns (region, tier, status) are usually dimensions.
Deduplication
Removing rows that are identical on a set of key columns. Essential before any aggregation when duplicate records are suspected.
sessionStorage
A browser mechanism for passing data between tabs in the same session. DataBridge uses it to hand off datasets to the Pricing Analyser without any server.
