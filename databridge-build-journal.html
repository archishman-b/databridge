Stage 0 · Origin
The access gap — where DataBridge came from
A specific problem, observed in a specific room, on a specific engagement
Grounded

During a pricing transformation engagement at Monitor Deloitte, the most important analysis we needed was structurally simple: group transactions by sales rep, compute average and median discount, sort descending, and identify the outliers. This is three SQL operations. It is roughly ten seconds of work for an analyst with access to the database.

It took 48 hours. The data was in SQL Server. We had no direct query access. The analyst pool was shared across multiple workstreams. Every iteration required a ticket, a handoff, a round trip. By the time the output arrived, the framing had shifted and we needed a different cut. Another 48 hours.

The founding observation
The bottleneck was never the analysis. It was the distance between the person who understood the business question and the person who could run the query. That distance has a name: the access gap.
— Pricing transformation, US auto parts retail · Monitor Deloitte

The same pattern appeared at Mahindra, where a consumer analytics team ran CLTV models and segment analyses on a 2M+ customer dataset in SQL Server. The strategic questions — which customer segments are growing, which are churning, which are high-value but under-served — were owned by the brand and channel teams. The analytical execution was owned by a dedicated analyst pool. Every question required a queue. Every iteration required another queue entry.

The tools that could close this gap — SQL, Python, Alteryx, Tableau Prep — all require either a licence, an installation, or a learning curve steep enough that most strategy practitioners never cross it. The gap persists not because better tools don't exist, but because the tools that close it are inaccessible to the people who need them most.

The design brief in one sentence Give strategy professionals direct access to the data operations they need — without code, without licences, without infrastructure — so that the person who has the question can also run the analysis.

Stage 1 · Problem Framing
What the access gap actually costs
Mapping the friction points with precision before designing anything
Done

Before building anything, the problem needed to be framed correctly. "Strategy professionals can't do data work" is too vague. The specific friction points are different, and they require different solutions.

The actual friction points
→Analyst queue lag — 24–72 hours per iteration on shared pools
→Infrastructure setup — SQL access requires DBA provisioning, VPN, credentials
→Tool licences — Alteryx, Tableau Prep, SPSS are per-seat enterprise buys
→Syntax dependency — even basic SQL requires learning a declarative language
→Iteration cost — every refinement to the question requires another round trip
What was NOT the problem
→Excel limitations — most strategy work fits in 50K rows; volume is rarely the issue
→Analytical complexity — the operations needed are join, filter, group, sort — not ML
→Visualisation — the output is a table for a slide, not an interactive dashboard
→Data storage — datasets are transient; nobody needs a database for one engagement
→Collaboration at scale — one person needs the answer, not a team of fifty

This framing is important. The solution space for "analysts can't do data work at scale" is a data warehouse. The solution space for "strategy professionals can't answer their own questions quickly" is a lightweight, self-contained, zero-infrastructure tool that runs the 15 operations that cover 90% of commercial analytics needs.

The scope boundary DataBridge is not trying to replace SQL Server, dbt, or Alteryx. It is trying to close the last mile — the gap between having a clean CSV export and having the answer to a specific business question. That last mile is 15 minutes of analyst work that costs 48 hours because of the queue.
Problem statement

Strategy professionals working on commercial analytics engagements — pricing, GTM, customer segmentation, post-M&A commercial architecture — routinely need to perform straightforward data transformations (filter, join, aggregate, segment) on datasets of 1,000–100,000 rows. These transformations require either SQL access (unavailable on most client engagements) or specialised tools (unlicensed or inaccessible). The result is a structural dependency on analyst pools for work that is not analytically complex, creating 24–72 hour delays per iteration that constrain the pace of strategic inquiry.

Stage 2 · Product Vision
Strategic bets & design philosophy
The four choices that define what DataBridge is and isn't
Done

A product vision is not just a description of the tool. It is a set of explicit bets — choices that have consequences, that foreclose other options, and that define what success looks like. DataBridge made four.

1
Zero infrastructure — the tool runs entirely in the browser
No server, no database, no authentication, no installation. A single HTML file that works offline, on any machine, with no setup time. Data never leaves the user's machine.
This is the only architecture that is actually accessible in every engagement context — client laptops, restricted networks, air-gapped environments, consultants working on trains.
2
Visual-first — no syntax, no expressions, no format to remember
Every operation is configured through dropdowns, pickers, and form inputs. The user never types a column reference or an operator. The interface encodes the grammar of data operations so the user doesn't have to learn it.
Syntax errors are the single largest source of friction for non-technical users. If a tool requires you to type [Revenue] > 1000 AND [Region] == 'North', it has not solved the access gap — it has just moved it one step to the right.
3
SQL as a learning layer — not a requirement
Every operation in DataBridge shows its SQL equivalent on demand. This is not required to use the tool — it is available for users who want to understand what they're doing at a deeper level, or who want to graduate to direct SQL access over time.
DataBridge's audience includes people who are analytically literate but not SQL-literate. The SQL companion bridges the gap without gating the tool behind it.
4
The data layer for the portfolio — not a standalone tool
DataBridge is designed to feed the Pricing Analyser and future analytical tools in the portfolio. The sessionStorage handoff protocol, the pricing schema mapper, and the named dataset architecture are all expressions of this role as the upstream data preparation layer.
The most valuable analytical tools are end-to-end workflows, not isolated operations. DataBridge → Pricing Analyser is a complete workflow from raw transaction export to boardroom-ready pricing diagnostic.

Stage 3 · Personas & User Stories
Who uses DataBridge and why
Four distinct users, one shared problem
Done

The access gap manifests differently for different users. Framing the personas precisely helps scope the operation set, the UX decisions, and the sequencing of backlog items.

Primary
The engagement consultant
MBA-trained, 3–8 years in strategy consulting. Comfortable in Excel, familiar with SQL concepts but no direct access. Works on client-provided data exports. Time-constrained and iteration-hungry.
Job to be done: "I have a client CSV and a question. I need the answer in the next 20 minutes without filing a ticket."
Primary
The in-house pricing manager
In-house commercial or pricing role. Has CRM or ERP exports but no BI team access. Runs monthly price review cycles. Needs repeatable analysis, not one-off scripts.
Job to be done: "I want to run the same discount-by-rep analysis every month without asking IT to pull a report."
Secondary
The strategy analyst
Junior analyst supporting a senior strategy team. Technically capable of writing SQL but doesn't have access on this engagement. Needs to move fast and produce clean output for slides.
Job to be done: "I need to join these two tables and group by region before my 3pm call. I can't wait for a BI request."
Secondary
The SaaS founder / head of revenue
Early-stage company, no dedicated data team. Has Stripe or CRM exports. Needs to understand cohort behaviour, segment revenue, and identify at-risk accounts without building a data stack.
Job to be done: "Show me which customers are at risk of churning based on their purchase history — without setting up a data warehouse."
Scope boundary
What DataBridge does not try to do DataBridge is scoped to datasets up to approximately 1M rows, operations that produce a table output (not a chart), and single-session analyses (no persistent storage of results). It is not a BI tool, not a dashboard builder, and not a replacement for a data warehouse. The tool knows its lane.

Stage 4 · Design Principles & Layout
How the interface encodes the principles
Every layout decision is a consequence of a design principle
Done

DataBridge has five design principles. They are not aesthetic preferences — they are constraints that resolve ambiguous design decisions.

P1
Zero syntax dependency
No user should ever need to type a column name, an operator, or an expression. Every configurable element is a dropdown, picker, or structured input. If a form field accepts free text, it is for a label or a value — never for a column reference or logical expression.
P2
Pipeline transparency
Every operation applied is recorded in the audit log with its parameters. The user can see exactly what has been done to the data, in what order, and undo any step. The pipeline is the analysis — it should be as readable as a bullet-pointed method section.
P3
SQL as optional depth
The SQL companion is always available but never required. Users who want to understand the mechanics can expand it; users who just want the answer ignore it. It never appears unless requested.
P4
Data quality before analysis
The quality scoring panel runs automatically on every data load. The user sees the quality score and issue summary before they run a single operation. This is not optional — a tool that lets you aggregate over a dataset with 40% nulls without warning is producing false confidence, not insight.
P5
Warm, legible, anti-intimidating
The visual aesthetic is deliberately warm and paper-like rather than the cold monochrome of engineering tools. The audience is strategy professionals who are not engineers — the interface should feel familiar and professional, not technical and foreign.
Three-zone layout

The layout encodes the workflow: tables and data loading on the left, operations panel on the right, and the result table in the centre. The topbar carries global actions (undo, audit log, export, PA handoff). The quality bar sits below the topbar and is only visible after data is loaded. The modal system handles all operation configuration without navigating away from the main view.

Layout decision: modal-based configuration Every operation is configured in a modal overlay rather than in an expanded panel or a dedicated page. This keeps the result table always visible in the background, so the user can see the current state of the data while configuring the next operation. Context is never lost.

Stage 5 · Architecture
Five decisions that determined everything
Architecture is product strategy expressed in code structure
Done
A1
Single HTML file — no build pipeline, no npm, no framework
DataBridge is one self-contained HTML file. All CSS, JavaScript, and HTML in one document. Deployable by dragging to GitHub Pages. Runnable by double-clicking a file. No dependencies to install, no bundler to configure, no build step to maintain.
Every additional dependency is a deployment barrier. A single file has one failure mode: the file doesn't load. That is a problem anyone can diagnose and fix.
A2
In-memory state — tables and pipeline in plain JavaScript objects
All data is held in tables (a dictionary of named datasets) and pipeline (an array of applied operations). There is no localStorage, no IndexedDB, no hidden state. The entire application state is readable in the browser console.
Debuggable, auditable, and private. Data never persists beyond the session — which is both a privacy feature and a simplicity feature.
A3
CDN-only dependencies — PapaParse and SheetJS only
DataBridge uses exactly two external libraries, both loaded from cdnjs. PapaParse for CSV parsing, SheetJS for XLSX reading. No charting library (DataBridge produces tables, not charts), no math library (operations use native JS), no UI framework.
Each additional CDN dependency is a network call, a version pinning decision, and a potential breaking change. Two libraries is the minimum viable set for the required functionality.
A4
Operations as pure functions — fn: data → data
Every operation in the pipeline is a pure function: it takes the current dataset as input and returns a transformed dataset. Operations do not modify state directly — they produce new state. This makes undo trivial (store the previous state), makes operations composable (chain any sequence), and makes the pipeline auditable (each step is an independent, reproducible transformation).
Pure functions are the simplest possible architecture for a pipeline system. They compose without side effects and require no coordination mechanism.
A5
sessionStorage handoff — the integration protocol with the Pricing Analyser
When the user clicks "→ Pricing Analyser", DataBridge serialises the current dataset and schema mapping to sessionStorage under the key pa_incoming_dataset. The Pricing Analyser reads this key on load and hydrates the dataset automatically. The session storage is cleared after hydration. No server, no API, no shared state beyond the browser session.
sessionStorage is the simplest possible inter-tab communication mechanism in a browser. It is synchronous, reliable within a session, and requires no infrastructure.

What v1.0 got wrong Five operations — filter, group-by, calculated column, conditional tagging, and rename — used free-text textarea inputs for configuration. This was a deliberate first-pass shortcut that immediately violated design principle P1. Users who loaded their own data found themselves needing to remember column names exactly as they appeared in the header row, type operator syntax correctly, and construct multi-condition expressions without autocomplete or validation. The access gap had not been fully closed. v1.1 fixed this.

Stage 7 · Build · v1.1
Visual builders, SQL companion, and the PA handoff layer
Completing the original vision and integrating with the Pricing Analyser
Live

v1.1 addressed the five free-text operations from v1.0, added the SQL companion panel, and shipped three backlog items (DB-01, DB-03, DB-04) that form the foundation of the DataBridge → Pricing Analyser integration workflow.

Fix 1 — Visual builders replacing all free-text inputs
Filter — rule builder
Visual rule rows: column picker + 11-option operator dropdown + value input. AND/OR toggle between rules. Add/remove rows. Logic connectors shown inline.
✓ v1.1
Group by — agg builder
Dynamic aggregation rows: column picker + SUM/COUNT/AVG/MIN/MAX/MEDIAN/COUNT DISTINCT + auto-named output column. Auto-suggests output name on column/function selection.
✓ v1.1
Calculated column
Three modes: Arithmetic (col A op col B), IF/THEN/ELSE (visual condition builder), Constant. Live formula preview updates as user configures.
✓ v1.1
Conditional tagging
Visual rule rows with label input. Default/else field at bottom. Evaluated top-to-bottom — first match wins. Add/remove rules freely.
✓ v1.1
Rename columns
Auto-generated one row per column. Current name shown as read-only. Editable new name field. No format to remember — just type the new name.
✓ v1.1
Fix 2 — SQL companion panel

A collapsed "⟨/⟩ View equivalent SQL" toggle at the bottom of every operation modal. Expanding it shows a read-only dark code block with syntax highlighting: keywords in blue, functions in green, strings in amber. The SQL is generated live from the current form state — it updates every time the user opens the panel. Covers all 40+ operations including the window function operations (RANK, ROLLING AVERAGE, RUNNING TOTAL, PERIOD COMPARISON).

Why the SQL companion matters beyond the immediate functionality DataBridge's stated purpose includes familiarising strategy professionals with the underlying data operations that analysts and engineers use. The SQL companion is how that learning happens — not through documentation or tutorials, but through seeing the exact SQL that produces the same output as the operation they just configured. The learning is contextual, optional, and immediate.
DB-01 — Pricing schema mapper

A guided column-mapping wizard accessible via the "◈ Pricing schema" button in the topbar after any data load. The wizard presents 12 standard pricing dimensions — list price, pocket price, unit cost, discount %, discount amount, quantity, sales rep, customer, customer tier, region, product, and date — and auto-suggests matches by scanning column names against regex patterns. The user confirms or corrects each mapping. The result is stored as window._pricingSchemaMap and included in the sessionStorage handoff payload.

DB-03 — Data quality scoring

A quality bar appears below the topbar after every data load. It shows a score from 0–100 (green ≥ 80, amber ≥ 60, red < 60) derived from four checks: null rate across all columns, duplicate row detection, mixed types in numeric columns, and negative values in money-named columns. Issue summaries are shown inline. A "Full null audit ↗" shortcut links directly to the null audit operation.

DB-04 — sessionStorage handoff to Pricing Analyser

The "→ Pricing Analyser" button appears in the topbar after data is loaded. Clicking it serialises the current dataset, column list, source name, row count, schema mapping, and timestamp to sessionStorage['pa_incoming_dataset'], then opens the Pricing Analyser in a new tab. The Pricing Analyser reads this key on load and hydrates the data automatically, bypassing its own upload and schema mapping wizard.

Sample data — v1.1

The inline sample data was replaced with a complete five-table demo dataset: 400 transaction rows (2023–2024), a 20-customer master, a 15-product catalog with pricing policy columns, a rep targets table with quota attainment, and a pre-joined PA-ready dataset with exact Pricing Analyser schema column names. Two reps — Emily Foster and David Evans — are engineered with 20–25% average discount (vs 7–9% for compliant reps), creating a visible leakage signal across all PA modules.

The remaining enhancement backlog, sequenced by the Pricing Analyser v2 dependency map
Planned

DB-01 through DB-04 were the critical path items for the Pricing Analyser v2 integration. The remaining backlog items are sequenced by their dependency on that integration and by their standalone value to DataBridge users.

DB-05
Multi-table join engine upgrade
Left/inner/outer joins on multiple key columns simultaneously. Sequential join support (join A to B, then join result to C). Match rate diagnostic shown after join.
Phase 2
DB-06
Date intelligence
Parse and normalise date columns across common formats. Extract period dimensions. Period-on-period comparison filter UI. Required for M3 trend view in Pricing Analyser v2.
Phase 2
DB-07
Multi-aggregate group-by
Multiple aggregation functions on the same column in one group-by operation (e.g. SUM and COUNT DISTINCT of revenue in the same step). Percentage-of-total as a native aggregation option.
Phase 2
DB-08
Large file streaming — Web Workers
Chunked processing for datasets above 100K rows. Progressive loading with row count displayed during parse. Hard warning at 100K, hard limit at 1M. Keeps the UI responsive on large files.
Phase 2
DB-09
Derived column builder — pricing metrics
One-click derived columns for standard pricing metrics: discount % from list and pocket price, margin % from revenue and cost, revenue per unit from revenue and quantity. Reads from the pricing schema mapping if set.
Phase 3
DB-10
SaaS metrics schema
Second schema profile alongside the pricing schema mapper: MRR, ARR, churn, expansion, cohort, tier, contract dates. Enables the Pricing Analyser's SaaS studio module to receive pre-mapped data from DataBridge.
Phase 3
DB-11
Outlier handling controls
Remove or winsorise rows above/below N standard deviations on any numeric column. Preview the impact (row count change, mean/median shift) before committing. Essential for clean regression inputs in M2.
Phase 4
DB-12
Saved schema profiles
Save a completed pricing or SaaS schema mapping as a named profile in localStorage. Auto-apply on next upload of a similarly-structured file (matched by column name similarity). Enables repeatable monthly analysis without re-mapping.
